# The platform-hub import leak stays reactively corrected — per-force visibility cannot filter the picker

Factorio 2.1's `LuaForce::set_script_visible` looked like the proactive fix for the documented hub-dropdown leak (every team's planet variants listed in the Import-from picker, corrected after the fact by `planet_map`'s reactive rewrite handler). Verification on 2.1.11 (headless API probes + in-game GUI test, 2026-07-21) established the engine facts: 2.1 couples `lock_space_location` ⇄ `is_visible` (locked → invisible per force), script overrides beat both tech-computed state and locks, survive `force.reset()` and `reset_technology_effects()`, and clear only with an explicit `nil` value — omitting the argument is an error despite the docs marking it optional. But the decisive GUI test failed both ways: on a claimed team, the hub's Import-from planet picker listed every team's variants and the base planets under lock-derived invisibility, and still did after an explicit bulk `set_script_visible(..., false)` over base nauvis and all foreign nauvis variants, re-checked after fully reopening the picker. The picker filters on neither axis.

Decision: the 2.1 opportunity (catalogued as A1) is **rejected**, and the reactive `on_entity_logistic_slot_changed` rewrite handler is the **permanent** mechanism — it already logs every correction, so its health stays observable. The lock choreography remains the single source of per-team location gating, and on 2.1 it *also* implies per-force invisibility — a side effect other GUI surfaces may respect even though this picker does not. Never set script-`true` on a foreign variant: it would present a visible but unschedulable location. The only remaining proactive path is a team-scoped picker inside MTS's existing hub relative GUI (opportunities doc B8), which stays deferred.

Resolved spike from the lock-invisibility coupling (RCON engine test, 2026-07-21): friends remote-view each other's variant surfaces from their own team force, and those locations are now invisible to it — but this does **not** break friend-view. A team force can set a remote-view controller onto a friend's surface whose space-location is invisible to it (`set_controller{type=remote, surface=…}` succeeds and renders), and MTS's own friend/spectator navigation uses precisely that path plus `set_surface_hidden`, never the vanilla space-map planet switcher (the only surface the invisibility touches). No friendship-visibility patch is required. The same reasoning covers consumer-registered real planets whose team surfaces are `associate_surface`'d to them (MTS Dimension Warp): the surface remains reachable regardless of the location's visibility.
